## What it does

SentinelOps is an automated incident response system that sits between your observability stack and your on-call team. It polls Elasticsearch every 30 seconds for logs and metrics, detects anomalies using statistical analysis, and uses Claude to reason about what's going wrong and how to fix it.

Instead of waking someone up for every noisy alert, SentinelOps triages incidents intelligently — deduplicating, correlating across services, and surfacing only what matters.

## How it works

1. **Monitor** — Continuously polls Elasticsearch for application logs and metrics across services.
2. **Detect** — Uses z-score statistical analysis to catch anomalies in error rates and p99 latency.
3. **Correlate** — Runs ES|QL queries to find cascading failures and related events across services.
4. **Analyze** — Sends context to Claude API to identify root causes and generate remediation steps.
5. **Prioritize** — Deduplicates alerts and assigns severity levels (P1–P4).
6. **Notify** — Sends structured Slack notifications (Block Kit) and pages PagerDuty for P1/P2 incidents.

## Tech stack

- **Python 3.11+** with asyncio for concurrent monitoring
- **Elasticsearch 8.x** as the data source
- **Claude API** (Anthropic) for intelligent analysis
- **Slack** (Block Kit) and **PagerDuty** for notifications
- **Pydantic v2** for data validation
- **Docker + Docker Compose** for deployment

## Why I built it

I wanted to explore how LLMs can be useful in operational contexts — not just generating text, but actually reasoning about system behavior. The goal was to reduce alert fatigue and give on-call engineers a head start on incident resolution by the time they look at an alert.
